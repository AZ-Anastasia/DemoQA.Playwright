## Что было выбрано в ходе разработки проекта для создания тестов для DemoQA

## UI
1. Был написан базовый класс для задания настроек и тестовых данных и подтягивания их из файла
2. Для отображения шагов в `Allure Report` (класс `AllureHelper`) был написан метод для скриншотов через `один из семейства паттернов Обертка (Wrapper) - Execute Around` для асинхронных и синхронного методов. Характерные отличия этого паттерна:
   - оборачивает процесс выполнения конкретной функции или делегата
3. Использовалась **декомпозиция** на **страницы** и **элементы** страниц
4. Для исключения повторного прописывания строковых значений из страниц и элементов (например, названия элементов аккордеона) использовались `перечисления`

## API

1. **Для клиентов** был выделен отдельный **базовый класс**, в котором из файла подтягиваются настройки
2. Был написан **метод расширения** для `IAPIResponse`\
для вызова по цепочке и для преобразования ответа сервера в **модель** из `Swagger`
3. Для передачи данных в метод `POST` был использован встроенный `Playwright DataObject`
4. Для повторного использования частей кода в тестах (например, добавление пользователя) были созданы тестовые шаги (`..Steps` в папке `TestsSteps`)
5. Из-за того, что `userId` можно получить от сервера только в методе `POST ../User`, приходится добавлять и удалять пользователя почти для каждого теста, а не использовать повторно
6. Чтобы не прописывать проверки тестов в классах, которые скрывают части шагов тестов (`..Steps`), было принято решение создать отдельный метод `EnsureSuccessAsync` для проверки выполнения запроса, чтобы убедиться, что запрос возвращает `Success` статусы
